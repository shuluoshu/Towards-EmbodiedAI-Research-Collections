0. 结论摘要

统一 MoT，但两种运行面

Cosmos 3 公开为两个 runtime surface：Reasoner 输入 text/vision 输出 text；Generator 输入 text/vision/sound/action 输出 vision/sound/action。本仓库 README 明确写 Cosmos 3 用统一 Mixture-of-Transformers 结构处理语言、图像、视频、音频和 action。

Action 是连续向量模态

公开 cookbook 和 cosmos-framework domain table 都显示 action 是连续数组：AV/camera 9D、hand/egocentric 57D、single-arm robot 10D、dual-arm 20D、AgiBot 29D。没有看到 VLM 离散 action token vocabulary。

DROID policy 默认 8D joint_pos

RoboLab policy server 默认服务 nvidia/Cosmos3-Nano-Policy-DROID，配置为 480p transform、15 FPS、32 action steps、8D joint_pos、guidance 3.0、4 denoising steps、shift 5.0。

关于 freeze：公开代码能直接确认的是「Reasoner AR decode 旁路 Generator multimodal embedders/heads」以及「RoboLab 推理使用 model.eval()」；公开 cookbook/可访问源码没有给出完整训练 recipe 中逐模块 requires_grad=False 的冻结表。因此本文把 freeze 分成“代码可证实”和“未公开/需训练配置验证”两类，避免把论文叙述误写成代码事实。

2. Backbone、参数量与 freeze/旁路关系

代码可证实：Reasoner AR decode 旁路 Generator 头

Cosmos3VFMNetwork.generate_reasoner_text 是对 language_model.generate_reasoner_text 的 thin pass-through。源码注释说明 Reasoner text generation 使用 non-generation pathway 权重以及 text embedding / norm / lm_head；VFM 级的 multimodal generation embedders/heads（如 vae2llm、llm2vae、sound heads 等）被 bypass。

代码可证实：Generator 不是三套 FD/ID/Policy head

Cosmos3VFMNetwork 的输出头按模态划分：vision 用 llm2vae，action 用 llm2action，sound 用 llm2sound。FD/ID/Policy 的区别体现在数据 batch 的 mode、conditioning/noisy masks 和输入输出组合，而不是源码中出现三套独立 fd_head/id_head/policy_head。

推理 eval，不等于训练 freeze

RoboLab policy server 加载模型后执行 self.model.eval() 并在 torch.inference_mode() 下调用 generate_samples_from_batch。这只是推理关闭 dropout/梯度，不代表训练阶段冻结了某些模块。

未公开/需验证：训练阶段逐模块 requires_grad

在可访问的 cookbook 和公开 raw 源码中，没有看到完整 DROID policy post-training recipe 的逐模块冻结表。因此本文不声称“某模块训练时必然冻结”，而是标注为：公开模型使用预训练 VLM/Generator backbone；Reasoner-only decode bypass generation heads；policy post-training 使用相同架构并输出 action。

3. Pre-training、Mid-training、Post-training 分别怎么训：代码层级逻辑

3.1 训练阶段总览

阶段训练内容代码层级上对应什么输出 checkpoint
Pre-training Reasoner 做视觉-语言 next-token；Generator 先学 image/video/audio generation。 Reasoner：AR text loss 路径；Generator：vision/sound latent noising + rectified-flow/denoising loss。Action mode 尚未作为核心数据行激活。 基础 Reasoner / Generator backbone。
Mid-training 加入 action、video transfer、Physical AI synthetic/domain 数据；形成跨 embodiment 的 world-action prior。 action_gen=True 激活 action2llm/llm2action；batch 中带 domain_id、raw_action_dim、mode、condition_mask/noisy indexes；action loss 与 vision loss 共同训练。 Cosmos3-Nano、Cosmos3-Super mid-trained base。
Post-training 对特定任务/domain 做专家化，例如 T2I、I2V、DROID policy。 沿用同一架构和 modality heads，换成高质量目标数据/任务模式；DROID policy 的 serving config 是 droid_lerobot、8D joint_pos、32-step action chunk。 Cosmos3-Super-Text2Image、Cosmos3-Super-Image2Video、Cosmos3-Nano-Policy-DROID。

3.2 Generator 数据 curriculum（论文级证据 + 代码入口）

technical report 摘要检索结果显示 Generator 训练使用 progressive multi-stage curriculum：pre-training 从 images/videos/audio 开始，mid-training 后加入 actions 和 interleaved multimodal content；mid-trained Nano/Super 再进入 post-training，得到 Text2Image、Image2Video、Policy-DROID 等专家模型。报告摘要还列出：pre-training image 767M、video 约 348M；mid-training 包含更高质量图像/视频、action 和 video transfer；post-training 各专家模型独立进行。

3.3 代码层级训练 forward：同一模型，多模态 flow matching

# 伪代码，来自 cosmos-framework 的结构而非逐字复制
packed = zeros([N_total, hidden])
packed[text_indexes]   = embed_tokens(text_ids)
packed[vision_indexes] = vae2llm(patchify(noisy_vision_latents)) + pos + timestep
packed[action_indexes] = action2llm(noisy_actions, domain_id) + pos + modality + timestep
packed[sound_indexes]  = sound2llm(noisy_sound_latents) + modality + timestep

hidden = shared_language_model(packed, packed attention metadata)
preds_vision = llm2vae(hidden[vision_noisy_indexes])
preds_action = llm2action(hidden[action_noisy_indexes], domain_id)
preds_sound  = llm2sound(hidden[sound_noisy_indexes])

loss = vision_flow_matching_loss + action_flow_matching_loss + sound_flow_matching_loss

关键点：action training 不是把 action 离散化成 tokenizer token；代码里 action 是 [T, action_dim] 连续张量，先由 DomainAwareLinear(action_dim → hidden) 投到 transformer hidden，再由 DomainAwareLinear(hidden → action_dim) 预测 velocity/denoised target。

4.3 DROID policy 默认 action：8D joint_pos

RoboLab observation

→

→
Cosmos model

generated action chunk
[32,8] before history trim; gripper flipped back

这与 midtrain 10D EEF delta 不同：DROID policy serving 的默认 action_space 是 joint_pos，raw action dim 是 8，即 7 个 joint positions + 1 个 gripper。server 文档明确说每个 request 需要 prompt、image、gripper_position，以及 joint state fields 或 EEF pose fields；默认 DROID policy 使用 joint state fields。

4.4 FD / ID / Policy 与 action 输出关系

Task mode输入生成目标与 policy 的关系
Forward Dynamics (FD)start image + action trajectoryfuture observations/video训练“动作导致什么视觉变化”的 world-action dynamics prior；不是最终机器人控制接口。
Inverse Dynamics (ID)video / visual transitionaction trajectory训练“视觉变化对应什么动作”的 action grounding；公开 cookbook 当前 ID 示例是 AV ego-motion，但接口可支持其他 domain 维度。
Policyinstruction + current image + state/historyfuture action chunk + optional rollout video最终控制机器人使用的闭环接口；共享 action head/backbone，而不是独立 policy-only MLP。

5. DROID policy 训练/推理路径：以实际例子说明

5.1 推理部署路径

# cosmos-framework 容器中
python -m cosmos_framework.scripts.action_policy_server_robolab --port 8000

# RoboLab client
python policies/cosmos3/run.py --task BananaInBowlTask

# 多环境 headless eval
python policies/cosmos3/run.py --task BananaInBowlTask --num-envs 10 --headless

本仓库 policy cookbook 说明 server 向 client stream actions，client 可以驱动模拟或真实机器人；示例 client 是 RoboLab，任务示例是 BananaInBowlTask。

5.2 Server 内部推理流程

  1. 解析 checkpoint，默认 nvidia/Cosmos3-Nano-Policy-DROID。
  2. 创建 OmniInference，拿到 pipe.model，调用 model.eval()。
  3. 构建 ActionTransformPipeline；如果 checkpoint/training config 可读，则从训练数据 config 推断 action chunk size、fps、resolution。
  4. 收到 OpenPI-style WebSocket observation：prompt、image、gripper、joint/eef state。
  5. 构造 sample：video 是 [3,T,H,W] 且第 0 帧为当前图像；action 是 [chunk + state_rows, action_dim]，第一行可放当前 state。
  6. 设置 mode='policy'、domain_id=get_domain_id('droid_lerobot')、viewpoint 和多视角描述。
  7. 调用 generate_samples_from_batch(... guidance, seed, num_steps, shift)。
  8. 取 samples['action'][0]，裁掉 history rows；默认 joint_pos 下翻转 gripper convention 后返回给 client。
  9. 如果 --decode-video，再从 samples['vision'] decode 预测 rollout video。

5.3 训练路径可见部分

公开 cosmos-framework 的 generic training entry 是 cosmos_framework.scripts.train，examples/toml 当前主要列出 vision/reasoner SFT 类 recipe；可访问的 examples 中没有完整 DROID policy post-training TOML。因此，这份报告把 DROID policy 训练解释为“模型代码支持的 action flow-matching + released checkpoint/serving config”，而不伪造未公开训练超参。

# 代码层级的 policy post-training 逻辑可抽象为：
for batch in droid_lerobot_demonstrations:
    sample = transform(video_frames, language_instruction, robot_state, action_trajectory)
    sample.mode = "policy"
    sample.domain_id = get_domain_id("droid_lerobot")
    noised_action, target_velocity = rectified_flow_action(action_trajectory)
    noised_vision, target_video_velocity = rectified_flow_vision(video_latent)
    preds = cosmos3_vfm_network(sample)
    loss = action_loss_weight * fm_loss(preds.action, target_velocity) + vision_loss_weight * fm_loss(preds.vision, target_video_velocity)
    update(shared_transformer + modality projections/heads according to training config)
最后一行的“哪些参数 update/freeze”必须以实际 DROID policy post-training recipe 为准；当前公开可访问源码没有给出逐参数冻结表。
